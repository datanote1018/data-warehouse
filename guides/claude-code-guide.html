是什么 + 安装配置 重点

1.1 Claude Code 是什么

Claude Code 是 Anthropic 推出的命令行 AI 编程代理，一个拥有完整系统访问权限的终端程序——能读写文件、执行命令、搜索代码、管理 Git、调用外部工具进行编码的智能体。

核心设计理念：不搞花哨的预处理，用最直接的方式完成任务。

很多 AI 编程工具（如 Cursor）在打开你的项目时，会先把所有代码建成索引数据库，你问问题时再从数据库里检索：

前期复杂但后续省 Token · 需要预处理 · 建索引 · 占存储 · 索引可能过时

Claude Code 不做这些——它每次启动时对你的项目一无所知，需要什么就直接用系统命令搜索，搜到了就读，读懂了就改，改完跑测试，没通过就继续搜、继续改，不断循环直到任务完成。

简单但费 Token · 零预处理 · 打开终端就能干活 · 搜索→理解→编码→验证→修正

就像一个新人工程师入职第一天：先翻目录看项目结构，再搜关键词找到相关代码，读懂之后自己写新代码，写完验证，有错就改。整个过程不需要任何预先准备，打开终端就能干活。

2.1 权限模式（Shift+Tab 切换）

Claude Code 通过 Shift + Tab 在不同权限模式间切换，控制 Claude 在多大程度上需要你确认才能执行操作。一共有 6 种模式，默认循环前 3 种，其他需要启用才出现：

权限模式行为适用场景默认可用
Default（Ask）每次操作都问你确认敏感操作、学习阶段是
Accept Edits自动允许文件编辑，其他仍需确认日常开发是
Plan只分析和规划，不动代码复杂任务先看方案是
Bypass Permissions完全跳过所有权限确认完全信任 AI需启用
AutoLLM 自动判断是否需要确认智能平衡需启用
Don't Ask不再询问批量操作需启用

从上到下，从最谨慎到最放开。也可以通过 settings.json 设置启动时的默认模式，或者用启动参数 claude --dangerously-skip-permissions 直接进入 Bypass 模式。

settings.json 配置权限规则：不管用哪种模式，都可以在 settings.json 里预设 allow/deny 规则，细粒度控制哪些命令自动放行、哪些必须拦截。 
{
  "permissions": {
    "allow": ["Bash(git *)", "Read(*)"],
    "deny": ["Bash(rm -rf*)", "Bash(sudo *)"]
  }
}
💡
实战经验
我用 claude --dangerously-skip-permissions 启动直接进入 Bypass 模式，配合 CLAUDE.md 里的规则做约束，日常使用基本无感。如果你在 CLAUDE.md 里定义了完整的工作流门控（比如"改代码前必须等用户确认"），Bypass 模式也不会乱来。

## 构建验证循环

P3

CLAUDE.md 与记忆管理系统 重点

这是 Claude Code 最核心的知识。Claude Code 的"记忆"和"行为"完全由文件控制。

3.1 三层配置架构

仅本地保留 提交到 Git（团队共享）
第一层：全局配置（所有项目生效） — 全部仅本地
~/.claude/CLAUDE.md ← 编码习惯、通用规则
~/.claude/commands/ ← 全局命令
~/.claude/skills/ ← 全局技能
~/.claude/settings.json ← 权限设置
第二层：项目配置（单个项目生效）
项目根/CLAUDE.md ← 技术栈、规范、架构 提交 Git
项目根/.claude/commands/ ← 项目命令 提交 Git
项目根/.claude/skills/ ← 项目技能 提交 Git
项目根/.claude/settings.json ← 项目权限 提交 Git
项目根/.claude/settings.local.json ← 本地权限 仅本地
第三层：对话/记忆（自动管理） — 全部仅本地
~/.claude/projects/{路径}/memory/MEMORY.md ← 记忆索引
feedback_*.md / project_*.md / user_*.md ← 分类记忆
~/.claude/projects/{路径}/sessions/ ← 会话存档

3.2 实战案例：DataNote 项目的完整配置

以下是一个真实项目的三层配置全貌，展示每一层应该放什么内容：

第一层：全局 CLAUDE.md — 跨项目通用规则
# Claude Code 全局规则

## 语言
所有回复和解释用中文。

## Agent 路由
搜索/翻译用 Sonnet，其他全部用 Opus。

## 行为准则
问问题先回答，改文件/写代码前必须等用户明确确认再动手。
全局 commands：notebooklm、yt-pipeline  |  全局 skills：youtube-search、wechat-article、skill-creator 等 7 个
第二层：项目 CLAUDE.md — 技术栈 + 工作流 + 红线规则
# DataNote 项目规则

## 技术栈约束
- Java 8（不用 var、text block、record）
- Spring Boot 2.7.18 + MyBatis-Plus 3.5.5
- 前端：单文件 SPA workspace.html（vanilla JS，不用框架）
- 数据库：MySQL 8.0 + Hive

## 默认工作流
收到开发任务时自动执行：
1. 需求分析 → 输出方案 → 等用户确认（确认前不写代码）
2. 开发实现
3. 代码自检（Java 8 兼容、循环依赖、SQL 注入、XSS）
4. 测试验证（编译 + 产物验证 + API curl 测试）
5. 5 步门控部署（停进程→编译→产物验证→启动→冒烟测试）
项目 commands：analyze、dbcheck、deploy、dispatch、frontcheck、jscheck、review、test  |  项目 agents：8 个对应的子智能体
第三层：记忆文件 — Claude 自动积累的经验
MEMORY.md（索引）
├── feedback_confirm_before_action.md — "问问题先答，改代码等指令"
├── feedback_datanote_restart.md     — "必须用 lsof 按端口杀旧进程"
├── feedback_ui_flow_order.md        — "表单按操作逻辑排列"
├── feedback_test_before_deliver.md  — "改代码前 grep 所有入口画调用链"
├── user_ui_aesthetic.md             — "干净、对齐、完整、克制的工程师审美"
├── project_docker_packaging.md      — "Docker 全栈打包方案"
└── reference_bilibili_api.md        — "B站 API 接口整理"
记忆会随着使用自动积累。每次犯错被纠正 → 存为 feedback；用户偏好 → 存为 user；项目决策 → 存为 project

P4

省 Token 的艺术：上下文管理与成本控制 重点

上下文窗口 = Claude 一次对话能"记住"的信息量。默认 20 万 Token，可扩展到 100 万（即 1M context，和 Max 订阅中的"1 兆上下文"是同一个概念）。

省 Token 最佳实践

  1. 新鲜上下文胜过臃肿上下文
  2. 勤用 /clear — 完成任务后清空
  3. 小任务分解 — 不要一次给巨大任务
  4. CLAUDE.md 不要太长 — 每条消息都重新注入
  5. MCP 不要贪多 — 每个 MCP 占上下文
  6. Plan → Clear → Execute — 规划后清空再执行
  7. 大文件建索引 — 项目中的大文件（如 8000 行的前端单文件）会占用大量上下文。正确做法是单独创建一个索引文件（如 index_workspace_html.md），记录每个模块的行号范围，然后在 CLAUDE.md 里用 @ 引用。这样索引不会每轮加载，只有 Claude 需要查这个大文件时才会去读索引，按需定位到具体行号范围，避免加载整个文件
  8. 用子 Agent 隔离上下文 — 把任务分给子 Agent 执行，子 Agent 结束后它的上下文就释放了，不会把搜索、分析过程中产生的大量中间结果堆到主会话里。主 Agent 只收到最终结果，上下文保持干净

/compact 手动压缩

实测从 24% 降到 13% Token 使用率。可附加提示词指定保留内容。

成本监控

Git 深度集成

Git 是比 Claude Code 内置回退更可靠的安全网。核心原则：尽早且频繁提交。

日常操作（个人开发者最常用）

"帮我提交这次修改，写好 commit message"
"推送到 GitHub"
"拉取最新代码"
"看一下最近的提交历史"

提交历史就是你的开发日志。

团队协作场景

  • PR 工作流：Claude 自动创建分支、推送、创建 PR
  • AI 代码审查：/review 和 /security-review
  • GitHub Actions：Issue → 自动修复 → 审查 → 合并 → 部署

P7

Hooks 自动化

Hooks = Claude Code 执行过程中自动触发的脚本。

Hook触发时机典型用途
pre-tool-use工具执行前拦截/替换命令
post-tool-use工具执行后自动格式化
user-prompt-submit提交提示词时预处理输入
session-start会话开始加载环境
stop任务完成自动跑测试

最强模式：Stop Hook + 自我纠错

{
  "hooks": {
    "stop": [{
      "command": "npm test",
      "on_failure": "continue"
    }]
  }
}

测试失败后 Claude 会继续修复直到通过。

P8

MCP 服务器：连接外部世界

MCP = Claude Code 连接外部工具的标准协议。

Server用途
Figma设计稿 → 代码
Playwright浏览器自动化
Context7最新编程库文档
GitHubIssue/PR 管理
DB Hub直接访问数据库
不要贪多！每个 MCP 都占上下文。10 个 MCP 不仅浪费 Token，还让 Claude 困惑。只装每天用的。

CLI 优先原则（官方）：如果工具既有 CLI 也有 MCP，优先用 CLI。MCP 的核心价值是给那些不能通过命令行访问的工具提供接入通道（如 Figma 设计稿、第三方 SaaS 平台）。如果命令行能搞定，就没必要多装一个 MCP。

P9

多 Agent 协作与子智能体 重点

Claude Code 本质是单智能体设计，通过子智能体扩展多 Agent 能力。

模型机制适用场景
Fork继承父上下文，缓存优化共享上下文的任务
Teammate独立终端，文件邮箱通信完全独立环境
Worktree独立 git 工作树并行开发不同功能

成本优势：子智能体共享提示词缓存（Prompt Cache），运行多个的成本远低于独立运行多个完整实例。
